##  Organization: .ipynb vs .py

### Where should each file go?

```
CORRECT project organization:

ai-finance-incidents/
│
├── notebooks/                   ← ALL .ipynb go here
│   ├── NB1_Exploration_Preparation.ipynb
│   ├── NB2_Statistical_Analysis.ipynb
│   ├── NB3_ML_Modeling.ipynb
│   ├── NB4_RESTful_API.ipynb
│   └── COMPLETE_UNIFIED_PROJECT.ipynb
│
├── app_api.py                   ← Flask API (Project ROOT)
│
└── dashboard/                   ← Dashboard in a separate subfolder
    ├── app.py                   ← Streamlit
    └── .streamlit/config.toml
```

### Why is `app_api.py` at the ROOT?

The Flask API file needs to find:
- `ai_finance_incidents.db` (generated by NB1 at the root)
- `models/*.pkl` (generated by NB3 at the root)

If it were in a subfolder, relative paths would break:

```python
# app_api.py — works at the root
DATABASE    = "ai_finance_incidents.db"       # ✅ found
MODELS_PATH = "models"                         # ✅ found

# If it were in /api/app.py — it would break
DATABASE    = "../ai_finance_incidents.db"     # ❌ fragile
MODELS_PATH = "../models"                      # ❌ fragile
```

### Why is `dashboard/app.py` in a separate SUBFOLDER?

**Real problem**: If both files were named `app.py` and placed in the same folder, running `streamlit run app.py` could cause Python to import the wrong one, causing a conflict.

**Adopted solution**: name the API `app_api.py` (or `api.py`) and the dashboard `dashboard/app.py`.

```
NEVER do:
project/
├── app.py   ← Flask API (confuses with the dashboard)
└── app.py   ← Streamlit Dashboard (impossible!)

CORRECT:
project/
├── app_api.py          ← Flask API
└── dashboard/app.py    ← Streamlit Dashboard
```

### Notebooks inside `notebooks/` (NOT at the root)

**Why?**
1. Keeps the root clean (only `app_api.py`, `requirements.txt`, `README.md`)
2. Separates analysis (notebooks) from production (Python code)
3. Makes it easier to ignore notebooks in the production `.gitignore`

### How to integrate notebooks with .py?

Notebooks **generate** the artifacts that Python code **consumes**:

```
Notebooks GENERATE:                 .py CONSUMES:
─────────────────────────────────────────────────────
NB1 → incidents_finance_filtered.csv  → dashboard/app.py (load_csv)
NB1 → ai_finance_incidents.db         → app_api.py (query_db)
NB3 → models/*.pkl                    → app_api.py (joblib.load)
NB3 → models/*.pkl                    → dashboard/app.py (via API)
NB4 → app_api.py                      → final project file
```

Python code **never imports** notebooks directly.

## Requirements: Separate or Unified?

### Recommendation: A SINGLE `requirements.txt` for this project

**Justification**:

```
CONTEXT            RECOMMENDATION     WHY
───────────────────────────────────────────────────────────────
Academic project   → Unified          Simplicity, maintenance
Portfolio          → Unified          Less friction for evaluators
Startup production → Separate         Deploy on distinct services
Microservices      → Separate         Independent containers
```

For an academic portfolio project, separate requirements create unnecessary complexity:

```bash
# With separate requirements — confusing for evaluators:
pip install -r requirements_api.txt
pip install -r requirements_dashboard.txt
pip install -r requirements_notebooks.txt
# Which one should I use? All? Just one?

# With unified requirements — simple and straightforward:
pip install -r requirements.txt
# Worked. Next step.
```

## Unified NB5 vs Separate Notebooks

### Full Analysis

| Criterion | Unified NB5 | Separate NB1–NB4 |
|---|---|---|
| **Reproducibility** | ⚠️ Shared global state | ✅ Each NB is isolated |
| **Debugging** | ❌ Hard to locate errors | ✅ Error isolated by phase |
| **Presentation** | ✅ One file, one click | ❌ 4 executions in the right order |
| **Collaboration** | ❌ Merge conflicts | ✅ Parallel work |
| **Readability** | ⚠️ 71 cells, can be confusing | ✅ Focus per phase |
| **Briefing/Evaluation** | ⚠️ Less structured | ✅ Clear CRISP-DM structure |
| **GitHub Portfolio** | ✅ One file to demonstrate | ✅ Professional structure |
| **Maintenance** | ❌ Editing one NB affects all | ✅ Independent edits |

### Final Recommendation

> **Use separate notebooks (NB1–NB4) as standard. Use NB5 only for presentation.**

**For development and academic delivery**:
```
NB1 → NB2 → NB3 → NB4 (mandatory order)
```
Reason: aligns with the CRISP-DM structure the briefing demands, and each notebook has clear deliverables.

**For a quick demonstration**:
```
COMPLETE_UNIFIED_PROJECT.ipynb (NB5)
→ "Run All" and done
```
Reason: evaluators and recruiters need less friction.

### Best Practices for Both Approaches

